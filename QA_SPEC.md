## 1. 온보딩 & 인증

**화면:** `app/onboarding.tsx`
**진입 조건:** 로그인되지 않은 상태

### 1.1 웰컴 화면 (step: welcome)

| 항목 | 상세 |
|------|------|
| 로고 이미지 | nyong_text.png 표시 |
| 앱 소개 문구 | 태그라인 + 설명 텍스트 |
| 개인정보 동의 체크박스 | 체크하지 않으면 구글 로그인 비활성화 |
| 개인정보 처리방침 링크 | 외부 브라우저에서 GitHub MD 파일 열림 |
| 구글 로그인 버튼 | 개인정보 동의 시에만 활성화, 로딩 스피너 표시 |
| 이메일 로그인 링크 | 개인정보 동의 후 step → signup 전환 |

**테스트 케이스:**
- [ ] 개인정보 동의 없이 구글 로그인 버튼 → 비활성화 상태 확인
- [ ] 개인정보 동의 체크 → 구글 로그인 버튼 활성화
- [ ] 개인정보 처리방침 링크 → 외부 브라우저 열림
- [ ] 구글 로그인 성공 → 닉네임 설정 또는 갤러리로 이동
- [ ] 구글 로그인 실패 → 에러 메시지 표시
- [ ] 개인정보 동의 없이 이메일 로그인 링크 → 동의 요청 알림
- [ ] 개인정보 동의 후 이메일 로그인 링크 → 회원가입 화면 전환

### 1.2 회원가입/로그인 화면 (step: signup)

| 항목 | 상세 |
|------|------|
| 이메일 입력 | TextInput, 이메일 형식 검증 (`user@domain.com`) |
| 비밀번호 입력 | secureTextEntry, 숫자·영문·특수문자 중 2가지 이상 조합, 5~20자 |
| 회원가입 버튼 | 이메일/비밀번호 검증 후 OTP 발송 |
| 로그인 버튼 | 기존 계정 로그인 |
| 뒤로 가기 | welcome으로 복귀 |

**테스트 케이스:**
- [ ] 빈 이메일/비밀번호로 가입 → 에러 메시지
- [ ] 이메일 형식 오류 (예: "abc", "abc@", "@domain") → "올바른 이메일 형식을 입력해주세요" 에러
- [ ] 비밀번호 4자 이하 → 에러 메시지
- [ ] 비밀번호 21자 이상 → 에러 메시지
- [ ] 비밀번호 숫자만 → 에러 (2가지 이상 조합 필요)
- [ ] 비밀번호 영문+숫자 6자 → 정상 통과
- [ ] 비밀번호 영문+특수문자 5자 → 정상 통과
- [ ] 유효한 이메일로 가입 → OTP 메일 발송 → verify 화면 전환
- [ ] 이미 가입된 이메일 → 에러 메시지 ("이미 가입된 이메일")
- [ ] 올바른 이메일/비밀번호로 로그인 → 갤러리 이동
- [ ] 잘못된 비밀번호 → 에러 메시지
- [ ] 존재하지 않는 이메일 로그인 → 에러 메시지

### 3.3 최신순 (latest) 보기

| 그리드 | 3열 |

| 잠금 사진 | 배달 후 1시간 경과 + 미열람 → 배달 날짜(M/D) + 🔒 아이콘 + 블러 |
| 일반 사진 | 탭 → 풀스크린 갤러리 진입 |
| 라벨 | 사진 하단에 뇽이름 + #태그 표시 |

**테스트 케이스:**
- [ ] 최신순 선택 → 최신 배달부터 표시
- [ ] 30장 이상 → 스크롤 끝에서 추가 로딩
- [ ] 새 사진 (status=delivered) → "NEW" 배지 표시
- [ ] 새 사진 탭 → 뇽펀치 화면 이동
- [ ] 1시간 이상 지난 미열람 사진 → 잠금 상태 + 날짜(M/D) 표시
- [ ] 잠긴 사진 탭 → 광고 확인 다이얼로그
- [ ] 광고 시청 후 → 잠금 해제 → 뇽펀치 화면
- [ ] 이미 본 사진 탭 → 풀스크린 갤러리
- [ ] 풀스크린에서 상하 스와이프 → 다른 사진 (세로 스크롤)
- [ ] 풀스크린에서 핀치 줌 → 확대/축소

### 3.6 서브갤러리 (뇽별 상세)

| 항목 | 상세 |
|------|------|
| 헤더 | ← 뒤로가기 + "{뇽이름} 갤러리" |
| 뇽 카드 | 대표 사진(원형) + ID 카드 버튼 + 이름 + 사진 수 |
| 사진 그리드 | 해당 뇽의 사진만 표시 |
| "+" 카드 | 그리드 마지막에 한장 더 받기 카드 |

#### "+" 카드 상태별 표시

| 상태 | 텍스트 | 동작 |
|------|--------|------|
| 무료 (usedToday=0) | "+ {뇽이름} 한장 더 받기" | 탭 → 즉시 사진 수신 → 뇽펀치 |
| 광고 필요 (usedToday=1) | "+ 광고보고 {뇽이름} 한장 더 받기" | 탭 → 광고 시청 → 사진 수신 → 뇽펀치 |
| 횟수 소진 (usedToday≥2) | "내일 또 받을 수 있어요" | 비활성 (opacity 0.5) |
| 사진 없음 (availablePhotos=0) | "더 받을 {뇽이름} 사진이 없어요" | 비활성 (opacity 0.5) |
| 기타 그룹 (nyongId=null) | - | 카드 미표시 |

**테스트 케이스:**
- [ ] 서브갤러리 진입 → 해당 뇽 사진만 표시
- [ ] ← 뒤로가기 → 뇽 그룹 목록 복귀
- [ ] ID 카드 버튼 → 뇽 ID 카드 화면
- [ ] "+" 카드 무료 탭 → 광고 없이 사진 수신 → 뇽펀치 이동
- [ ] 뇽펀치 후 복귀 → 서브갤러리에 새 사진 반영 + 사진 수 갱신
- [ ] "+" 카드 두 번째 → 광고 시청 → 사진 수신
- [ ] 광고 스킵/실패 → 사진 미수신
- [ ] 세 번째 탭 → "내일 또 받을 수 있어요" 비활성
- [ ] A뇽에서 무료 1장 사용 → B뇽에서는 "광고보고..." 표시 (글로벌 카운트)
- [ ] B뇽에서 광고 1장 사용 → 모든 뇽에서 "내일 또 받을 수 있어요"
- [ ] 사진 풀 소진된 뇽 → "더 받을 {뇽이름} 사진이 없어요"
- [ ] 기타 그룹에서는 "+" 카드 미표시
- [ ] 풀에는 정기 배달된 적 있는 사진만 포함 (오늘 업로드/미래 예약 제외)

## 4. 뇽펀치 (알림/펀치 화면)

### 4.1 펀치 전 (showHint=true)

| 항목 | 상세 |
|------|------|
| 상단 | ← 뒤로가기 버튼 |

### 5.5 매칭 (catMatcher)

| 항목 | 상세 |
|------|------|
| 방식 | 텍스트 임베딩 코사인 유사도 (text-embedding-3-small) + GPT-4o 특징 비교 |
| ≥0.55 | 확정 매치 → 업로드 허용 |
| 0.20~0.55 | GPT-4o 추가 확인 → 털색/품종 수준만 체크 (관대) |
| <0.20 | 확실히 다른 고양이 → 업로드 차단 |
| 차단 조건 | `!isMatch && confidence < 30`일 때만 차단 (매우 관대) |
| 매칭 에러 | 매칭 서버 오류 시 **업로드 허용** (사용자 불이익 방지) |

> ※ 등록 시: extractCatFeatures 텍스트 → 임베딩 (GPT 호출 절약)
> ※ 업로드 시: gpt-4o-mini 이미지 설명 → 임베딩
> ※ 비교는 "같은 고양이인가"가 아니라 "털색/품종이 비슷한가" 수준 (가드레일)

**테스트 케이스:**
- [ ] 뇽 미선택 → "뇽을 선택해주세요" 에러
- [ ] 사진 미선택 → 업로드 버튼 비활성화
- [ ] 고양이 아닌 사진 → "고양이 사진이 아닌 것 같아요" → 업로드 차단
- [ ] 같은 뇽에 오늘 두 번째 업로드 → "이미 업로드했어요"
- [ ] 확실히 다른 고양이 (유사도 <0.20, confidence <30) → 차단
- [ ] 털색/품종 비슷한 고양이 (유사도 0.20~0.55) → GPT-4o 추가 확인 → 관대하게 허용
- [ ] 동일 고양이 (유사도 ≥0.55) → 즉시 매칭 성공
- [ ] 뒷모습/옆모습/잠자는 포즈 → 털색 비슷하면 매칭 통과
- [ ] 매칭 서버 오류 → 업로드 허용 (경고 로그만 기록)
- [ ] 정상 업로드 → 성공 메시지 + 포인트 +1
- [ ] 태그 입력 → 사진에 태그 저장
- [ ] 업로드 중 스피너 표시
- [ ] 업로드 성공 후 UI 리셋

- [ ] 빈 상태 → 안내 메시지 + (본인 뇽이면) 업로드 버튼
- [ ] 명예의 전당에서 진입 → 정상 표시

### 8.1 Hits/통계 데이터 정확성

**데이터 소스:** `get_nyong_uploads` RPC (SECURITY DEFINER) + `get_top_puncher` RPC

| 항목 | 상세 |
|------|------|
| 사진별 hits | `SUM(deliveries.hits)` — 해당 upload의 모든 delivery hits 합산 |
| 총 뇽펀치 | 클라이언트에서 모든 upload의 hits를 reduce로 합산 |
| 월간 펀치 | 클라이언트에서 모든 upload의 hits를 reduce로 합산 (DB monthly_hits 미사용) |
| Top 3 펀처 | `get_top_puncher` RPC — upload_id별 receiver의 hits 상위 3명 |

**테스트 케이스 (DB 검증):**
- [ ] DB: `get_nyong_uploads` 반환값의 hits == `SELECT SUM(d.hits) FROM deliveries d WHERE d.upload_id = ?`
- [ ] DB: 여러 수신자가 같은 upload에 뇽펀치 → 모든 수신자의 hits 합산
- [ ] DB: extra delivery로 생성된 배달의 hits도 합산에 포함
- [ ] DB: nyong_extra delivery로 생성된 배달의 hits도 합산에 포함
- [ ] DB: sender_id가 실제 업로더(user_id)와 일치하는지 확인 (과거 버그: extra delivery에서 sender_id = receiver_id로 잘못 설정됨)

**테스트 케이스 (RLS/크로스유저):**
- [ ] 업로더(A)가 본인 뇽 갤러리 → 모든 upload의 hits 합산 표시 (RPC가 RLS 우회)
- [ ] 다른 사용자(B)가 명예의 전당에서 A의 뇽 갤러리 → 동일한 hits 표시
- [ ] 다른 사용자(B)가 업로드 히스토리에서 A의 뇽 선택 불가 (업로드 화면은 본인 뇽만)

**테스트 케이스 (Top 펀처):**
- [ ] get_top_puncher: hits > 0인 delivery만 포함 (hits=0은 제외)
- [ ] get_top_puncher: 최대 3명만 반환
- [ ] get_top_puncher: hits 내림차순 정렬
- [ ] 롤링 펀처 닉네임이 프로필에서 변경 후 → 갤러리 새로고침 시 업데이트

**테스트 케이스 (엣지 케이스):**
- [ ] 배달 0건인 upload → hits = 0 표시
- [ ] 모든 delivery가 hits=0인 upload → hits = 0 표시
- [ ] uploaded_at이 미래인 upload → `get_nyong_uploads`에서 제외 (NOW() 필터)

### 13.4 배달 데이터 정확성 (DB 검증)

**sender_id 무결성:**
- [ ] 정기 배달 (scheduled): `deliveries.sender_id` = `uploads.user_id` (업로더의 UUID)
- [ ] 하나 더 받기 (extra): `deliveries.sender_id` = `uploads.user_id` (업로더의 UUID, ≠ receiver_id)
- [ ] 뇽별 한장 더 (nyong_extra): `deliveries.sender_id` = `uploads.user_id`
- [ ] DB 전수 검사: `SELECT * FROM deliveries d JOIN uploads u ON d.upload_id = u.id WHERE d.sender_id <> u.user_id` → 결과 0건

**source 필드 정확성:**
- [ ] 정기 배달 → `source = 'scheduled'`
- [ ] `get_extra_delivery` RPC → `source = 'extra'` (또는 NULL 여부 확인)
- [ ] `get_nyong_extra_delivery` RPC → `source = 'nyong_extra'`
- [ ] DB: source가 NULL인 delivery 존재 여부 확인

**status 라이프사이클:**
- [ ] 신규 배달 → `status = 'delivered'`, `hits = 0`, `received_at = NULL`
- [ ] 뇽펀치 완료 → `status = 'received'`, `hits > 0`, `received_at IS NOT NULL`
- [ ] record_delivery_hits RPC: `status = 'delivered'`인 건만 업데이트 (중복 방지)
- [ ] record_delivery_hits RPC: 이미 `status = 'received'`인 건 → 무시 (히트 중복 적립 방지)

**record_delivery_hits RPC 원자성:**
- [ ] deliveries.hits 업데이트 + nyongs.total_hits 증가 + nyongs.monthly_hits 증가 → 트랜잭션으로 일괄 처리
- [ ] RPC 도중 오류 발생 → 부분 업데이트 없음 (rollback)
- [ ] 동시 호출 (race condition): 같은 delivery에 대한 동시 record_delivery_hits → 데이터 일관성 유지

**get_extra_delivery RPC:**
- [ ] 어제 업로드 사진 중 본인 업로드 제외 (`user_id <> receiver_uuid`)
- [ ] 이미 받은 사진 제외 (`NOT EXISTS deliveries WHERE upload_id AND receiver_id`)
- [ ] 랜덤 선택 (`ORDER BY RANDOM() LIMIT 1`)
- [ ] 배달 레코드 생성: `sender_id = uploads.user_id` (업로더), `receiver_id = receiver_uuid`
- [ ] 풀에 사진 없을 때 → 빈 결과 반환 (에러 아님)

**get_nyong_extra_delivery RPC:**
- [ ] 하루 2회 제한 (전체 뇽 합산): `used_today < 2`
- [ ] 정기 배달 이력 필수: `source = 'scheduled'`인 delivery 1건 이상
- [ ] 본인 업로드 제외
- [ ] 미수신 사진만 선택
- [ ] nyong_extra_usage 카운터 업데이트

**find_eligible_receiver RPC:**
- [ ] 오늘 이미 배달 받은 사용자 제외
- [ ] 방해금지 사용자 제외 (현재 시각이 DND 범위일 때)
- [ ] 자정 넘김 DND (22:00→08:00) 정상 처리
- [ ] 랜덤 선택 (`ORDER BY RANDOM() LIMIT 1`)

## 17. DB/RLS/Server-side 무결성 테스트

> 이 섹션은 클라이언트 UI 테스트로는 발견할 수 없는 서버사이드 데이터 무결성, RLS 정책, RPC 함수 동작을 검증합니다.
> DB 쿼리를 직접 실행하여 검증해야 합니다.

### 17.1 RLS 정책 크로스유저 테스트

**uploads 테이블:**
- [ ] 사용자A가 `SELECT * FROM uploads` → 본인 uploads만 반환 (RLS: `auth.uid() = user_id`)
- [ ] 사용자A가 사용자B의 upload_id로 `SELECT` → 결과 없음
- [ ] 사용자A가 `user_id = B의UUID`로 INSERT 시도 → RLS 위반
- [ ] SECURITY DEFINER RPC (`get_nyong_uploads`)로는 다른 사용자 uploads도 조회 가능

**deliveries 테이블:**
- [ ] 업로더(sender)가 `SELECT * FROM deliveries WHERE sender_id = auth.uid()` → 본인이 sender인 건만 반환
- [ ] 수신자(receiver)가 `SELECT * FROM deliveries WHERE receiver_id = auth.uid()` → 본인이 receiver인 건만 반환
- [ ] 사용자A가 sender도 receiver도 아닌 delivery → 조회 불가
- [ ] 업로더가 본인 사진의 delivery를 보려면 `sender_id = 업로더UUID`여야 함 → 과거 버그에서 sender_id가 receiver로 설정되면 업로더가 조회 불가

**nyongs 테이블:**
- [ ] 누구나 `SELECT * FROM nyongs` → 모든 뇽 조회 가능 (공개 읽기)
- [ ] 사용자A가 사용자B의 뇽에 `UPDATE` → RLS 위반 (`auth.uid() = owner_id`)
- [ ] 사용자A가 사용자B의 뇽에 `DELETE` → RLS 위반

**profiles 테이블:**
- [ ] 사용자A가 `SELECT * FROM profiles WHERE id = B의UUID` → 결과 없음 (본인만)
- [ ] 사용자A가 사용자B의 프로필 `UPDATE` → RLS 위반

**content_reports 테이블:**
- [ ] 사용자A가 `reporter_id = B의UUID`로 INSERT → RLS 위반
- [ ] 사용자A가 본인이 아닌 신고 `SELECT` → 결과 없음

**nyong_extra_usage 테이블:**
- [ ] 사용자A가 다른 사용자의 usage 조회 → 결과 없음

### 17.4 데이터 일관성 정기 검증 쿼리

> 운영 중 주기적으로 실행하여 데이터 이상을 탐지하는 쿼리 목록

```sql
-- 1. sender_id ≠ uploads.user_id인 배달 찾기 (과거 버그 재발 감지)
SELECT d.id, d.sender_id, u.user_id, d.source
FROM deliveries d
JOIN uploads u ON d.upload_id = u.id
WHERE d.sender_id <> u.user_id;

-- 2. nyongs.total_hits vs 실제 SUM(deliveries.hits) 불일치 확인
SELECT n.id, n.name, n.total_hits,
  COALESCE(SUM(d.hits), 0) AS actual_hits
FROM nyongs n
LEFT JOIN uploads u ON u.nyong_id = n.id
LEFT JOIN deliveries d ON d.upload_id = u.id
GROUP BY n.id, n.name, n.total_hits
HAVING n.total_hits <> COALESCE(SUM(d.hits), 0);

-- 3. birthday 형식 검증 (YYYY-MM-DD)
SELECT id, name, birthday FROM nyongs
WHERE birthday IS NOT NULL
  AND birthday::text !~ '^\d{4}-\d{2}-\d{2}$';

-- 4. front_embedding 차원 검증
SELECT id, name, array_length(front_embedding, 1) AS dim
FROM nyongs
WHERE front_embedding IS NOT NULL
  AND array_length(front_embedding, 1) <> 512;

-- 5. features JSON 키 누락 확인
SELECT id, name FROM nyongs
WHERE features IS NOT NULL
  AND NOT (
    features ? 'fur_color'
    AND features ? 'fur_pattern'
    AND features ? 'eye_color'
    AND features ? 'ear_shape'
    AND features ? 'distinctive_marks'
    AND features ? 'body_type'
  );

-- 6. status='delivered'인데 received_at이 있는 비정상 건
SELECT id, status, received_at, hits FROM deliveries
WHERE status = 'delivered' AND received_at IS NOT NULL;

-- 7. status='received'인데 hits=0인 건 (펀치 없이 완료?)
SELECT id, upload_id, receiver_id FROM deliveries
WHERE status = 'received' AND hits = 0;

-- 8. 같은 upload + 같은 receiver 중복 배달 확인
SELECT upload_id, receiver_id, COUNT(*) AS cnt
FROM deliveries
GROUP BY upload_id, receiver_id
HAVING COUNT(*) > 1;

-- 9. nyong_extra_usage 날짜별 2회 초과 사용 확인
SELECT user_id, use_date, SUM(count) AS total
FROM nyong_extra_usage
GROUP BY user_id, use_date
HAVING SUM(count) > 2;

-- 10. orphan deliveries (upload 삭제됨)
SELECT d.id FROM deliveries d
LEFT JOIN uploads u ON d.upload_id = u.id
WHERE u.id IS NULL;
```

### 17.5 멀티유저 시나리오 테스트

**시나리오 1: 업로더 → 수신자 → 업로더 확인**
